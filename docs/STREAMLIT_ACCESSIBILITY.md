Streamlit accessibility notes

Summary
- After in-app fixes (radio semantics, higher-contrast tokens, KaTeX tabindex removal), automated axe-core still reports a small number of violations primarily caused by Streamlit's generated DOM and widgets.

Remaining violations (observed)
- `aria-required-attr` (critical): Streamlit `select`/`combobox` inputs expose `role="combobox"` but do not include every ARIA attribute axe expects (e.g., `aria-expanded`, `aria-controls`). We add a best-effort attribute via injected JS, but Streamlit re-renders may overwrite it.
- `color-contrast` (serious): A handful of caption/text nodes reported low contrast by axe in some contexts. In practice we've increased our `--muted` color and set caption text to use `--ink` to maximize contrast; remaining flags are inconsistent across Streamlit themes and appear to be edge cases where axe measures against computed backgrounds.
- `landmark-banner-is-top-level` / `landmark-main-is-top-level` (moderate): Streamlit places header/landmark elements at app-level; attempts to add/remove `role` attributes can cause axe to flag duplicates. Many of these are intrinsic to Streamlit's app shell.
- `scrollable-region-focusable` (moderate): Some KaTeX-generated display blocks are focusable; we remove `tabindex` where possible, but third-party renderers and Streamlit internals can reintroduce focusable regions.

Why these are not fully fixable in-app
- Streamlit controls (selects, date pickers, sidebar) render complex DOM structures and manage ARIA attributes internally. Overwriting or patching these is brittle: Streamlit may re-render and reintroduce the original attributes.
- Some violations reference top-level shell semantics (header, main) that are outside the app's content area and are generated by Streamlit itself.

Recommendations
- Document these as framework-level limitations in the project's accessibility section and in PRs so reviewers understand the scope.
- Open an upstream issue with Streamlit providing the axe violation report and examples; Streamlit maintainers may accept fixes at the framework level.
- Keep in-app fixes minimal and focused on elements we control:
  - Provide correct ARIA semantics for our custom UI (`role=radio`, `aria-checked`, `aria-label` for choice blocks).
  - Ensure color tokens meet WCAG contrast (we now set `--muted` darker and caption text to `--ink`).
  - Strip `tabindex` from KaTeX display elements to avoid scrollable-region-focusable violations.
- In CI, run axe but only fail PRs on violations within our app namespace (e.g., nodes matching `.bm-*, .stApp .our-app-namespace`).

Notes for maintainers
- The most actionable next step upstream is to request Streamlit ensure combobox roles include `aria-expanded` and `aria-controls` and to reduce unexpected `role` attributes on `body`.
- Our injected JS is a pragmatic workaround but should be considered fragile. Prefer framework fixes when available.
